## Pendahuluan

Platform yang kami kembangkan ialah Inti Rupa, yang merupakan sebuah sistem asisten cerdas berbasis cloud yang dirancang untuk mengintegrasikan fitur analisis teks (Summarizer) dan kreativitas visual (AI Image Generator) berbasis arsitektur microservices. Dalam arsitektur sistem terdistribusi, aspek keandalan (fault tolerance), visibilitas internal sistem (observability), serta pengerasan keamanan (security hardening) menjadi prioritas utama guna menjamin kestabilan platform saat memproses permintaan komputasi AI yang intensif dari pengguna.

Dalam pengembangan proyek Inti Rupa ini, saya memegang peran sebagai Lead QA & Docs. Tanggung jawab saya mencakup penjaminan mutu, validasi perilaku sistem, hingga penyusunan dokumentasi operasional seluruh infrastruktur—memastikan bagaimana setiap komponen, mulai dari Nginx API Gateway, Auth Service, hingga AI Service beserta basis datanya masing-masing, dapat berfungsi secara andal. Secara spesifik, saya bertanggung jawab memastikan bahwa mekanisme penanganan kesalahan seperti Circuit Breaker, Exponential Backoff Retry, format Structured Logging JSON, hingga proteksi Rate Limiting berjalan sesuai spesifikasi teknis dan lulus uji kelayakan produksi.

## Kontribusi dalam Proyek

Sebagai Lead QA & Docs, kontribusi saya berfokus pada perancangan matriks pengujian fungsional dan penyusunan dokumentasi teknis yang komprehensif dari hulu ke hilir. Langkah awal yang saya lakukan adalah menyusun dan mengeksekusi skenario uji keandalan dari RT-01 hingga RT-08, yang kemudian saya tuangkan ke dalam berkas docs/reliability-testing.md. Saya memvalidasi fungsionalitas sakelar otomatis Circuit Breaker dan pembatasan waktu tunggu (Timeout) saat mendeteksi simulasi pemadaman paksa kontainer dependensi melalui perintah docker compose kill.

Untuk mendukung keterbacaan sistem, saya melakukan verifikasi terhadap alur Distributed Tracing dengan menguji aliran Correlation ID dari Gateway hingga ke seluruh layanan internal, serta menyusun berkas docs/operations-guide.md untuk memandu tim dalam melakukan pelacakan log secara taktis. Di sisi keamanan siber, saya memimpin audit kredensial dan melakukan verifikasi penapisan berkas rahasia .env pada Git. Seluruh rangkaian penjaminan mutu ini ditutup dengan pembuatan gerbang pemeriksaan kualitas akhir pada berkas docs/final-checklist.md serta memperbarui seluruh dokumentasi teknis pada halaman utama README.md guna memastikan arsitektur microservices tim telah memenuhi kriteria Production Readiness sebelum rilis final versi v3.0.0.

## Tantangan yang Dihadapi

Selama proses penjaminan mutu, terdapat beberapa tantangan teknis yang cukup kompleks dan memerlukan penyelesaian taktis di lingkungan terminal lokal. Salah satu hambatan utama terjadi akibat ketidaksesuaian sintaksis antara instruksi laboratorium berbasis Linux (bash) dengan perangkat pengujian lokal saya yang berbasis Windows PowerShell. Ketika mengeksekusi perintah pengujian otomatis, penggunaan parameter Linux seperti -s disalahartikan oleh PowerShell sebagai argumen SessionVariable yang tidak lengkap, sehingga memicu error berulang kali pada konsol terminal. Kondisi ini memaksa saya untuk memetakan ulang perintah-perintah tersebut menjadi skrip pengujian native PowerShell terbungkus blok try-catch agar kode respons HTTP dapat ditangkap secara valid.

Tantangan berikutnya muncul dari ketatnya skema validasi data (Pydantic) yang diterapkan oleh tim Backend. Saat menguji alur registrasi akun pengujian, saya berulang kali menemui respons 422 Unprocessable Entity karena payload JSON dari terminal tidak sinkron dengan kebutuhan model backend, seperti pengiriman kolom name yang seharusnya diwajibkan menggunakan kolom username dan full_name. Selain itu, pengelolaan status (state) Docker juga menjadi kendala tersendiri; beberapa kontainer dependensi sering kali tertinggal dalam kondisi mati akibat efek skenario uji fungsional sebelumnya, atau variabel token JWT lokal pada sesi PowerShell terhapus di tengah pengujian, sehingga alur autentikasi harus diulang dari awal.

Hambatan serupa terjadi saat memvalidasi Rate Limiting pada Nginx Gateway. Tingginya efisiensi pemrosesan internal backend dalam melakukan hashing password membuat hantaman request dengan jeda waktu selalu direspons dengan status 401 Unauthorized tanpa memicu batas burst 429 Too Many Requests. Saya harus mengubah strategi pengujian dengan memodifikasi skrip menjadi hantaman agresif bervolume 30 request instan tanpa jeda agar ambang batas keamanan Gateway benar-benar teruji secara objektif.

## Pembelajaran yang Didapat
Melalui proyek Inti Rupa ini, saya memperoleh pemahaman mendalam mengenai pentingnya transparansi sistem terdistribusi melalui structured logging berformat JSON dan pelacakan Correlation ID lintas layanan. Saya menyadari bahwa peran penjaminan mutu tidak sekadar mencari kesalahan kode, melainkan memahami batas toleransi sistem (SLA Thresholds) dan memastikan fitur darurat seperti Graceful Degradation dapat aktif untuk menjaga kenyamanan user experience saat krisis jaringan terjadi.

Dalam hal troubleshooting, saya belajar cara mendiagnosis anomali infrastruktur secara objektif melalui analisis log forensik dan pembacaan kode status HTTP (seperti membedakan kegagalan transient dengan kesalahan input deterministik) daripada sekadar menebak letak kegagalan komponen.

Secara non-teknis, peran ini membentuk mentalitas tanggung jawab yang besar sebagai Gatekeeper kualitas rilis produk. Ketika dokumentasi kontrak API, skrip verifikasi, atau panduan operasional yang saya susun tidak akurat, maka pekerjaan anggota tim lain—seperti sinkronisasi komponen oleh tim frontend, perbaikan kode oleh backend, hingga penyusunan slide presentasi UAS—akan langsung terhambat. Tantangan ini melatih saya untuk tetap tenang di bawah tekanan integrasi sistem yang masif, berpikir taktis, dan mengambil keputusan dokumentasi yang efisien demi kesuksesan proyek akhir tim.
