## Arsitektur dan Keputusan Teknis
Sebagai Lead DevOps, tanggung jawab utama saya adalah merancang infrastruktur yang dapat mendukung arsitektur *microservices* secara andal. Keputusan teknis pertama yang saya buat adalah mengadopsi **Docker dan Docker Compose** sebagai fondasi orkestrasi lokal. Pendekatan ini dipilih bukan sekadar untuk kemudahan instalasi, melainkan untuk menjamin paritas lingkungan (*environment parity*) antara fase pengembangan dan produksi, yang secara efektif meminimalisir anomali klasik "it works on my machine".

Selain itu, saya memutuskan untuk menempatkan **Nginx sebagai API Gateway**. Alih-alih mengekspos setiap layanan (*frontend*, `auth-service`, `ai-service`) secara langsung ke publik, Nginx bertindak sebagai *reverse proxy* terpusat. Keputusan arsitektural ini sangat krusial karena memungkinkan saya untuk menangani manajemen *CORS* (Cross-Origin Resource Sharing) dan *routing* di satu titik (*single point of entry*), sekaligus meningkatkan lapisan keamanan dengan mengisolasi komunikasi *database* dan *backend* hanya di dalam jaringan internal Docker.

## Kesulitan Teknis dan Resolusi
Proses transisi menuju otomatisasi dan infrastruktur riil menghadirkan tantangan teknis yang signifikan. Kurva pembelajaran paling tajam terjadi pada **orkestrasi jaringan antar-kontainer** dan **konfigurasi Continuous Deployment (CI/CD)**. 

Pada awalnya, mengonfigurasi rute Nginx agar dapat merutekan API dengan tepat sambil menjaga integrasi *frontend* sering kali memicu hambatan *CORS* dan *Bad Gateway* (502). Menyelesaikan isu ini menuntut saya untuk menganalisis bagaimana *header* HTTP diteruskan dari *client* ke *gateway* lalu ke *upstream services*. 

Tantangan kedua adalah merancang *pipeline* CI/CD untuk melakukan *deployment* otomatis ke platform **Railway**. Memastikan *build context* dari setiap `Dockerfile` (frontend, auth, dan AI) tepat sasaran dan mengamankan injeksi *environment variables* di lingkungan CI membutuhkan banyak iterasi. Kegagalan *build* di *pipeline* mengajarkan saya betapa ketatnya perbedaan lingkungan eksekusi antara *runner* GitHub/Railway dengan mesin lokal.

## Pelajaran Berharga: Esensi Observabilitas
Proyek ini menggeser paradigma saya dari sekadar "menulis kode agar berjalan" menjadi "bagaimana sistem dikemas, didistribusikan, dan diawasi". Saya menyadari bahwa *deployment* bukanlah akhir dari tanggung jawab DevOps.

Kesadaran ini mendorong saya untuk mengimplementasikan mekanisme **Monitoring dan Observabilitas** dasar menggunakan modul `stub_status` pada Nginx. Dengan mengekspos metrik lalu lintas API Gateway, saya memfasilitasi visibilitas yang lebih transparan untuk *dashboard* status aplikasi. Pelajaran terbesar yang saya bawa dari peran ini adalah bahwa DevOps bukan sebatas penguasaan *tools* (seperti Docker atau GitHub Actions), melainkan seni membangun sistem yang *resilient* (tangguh), dapat diamati, dan meminimalkan intervensi manual melalui otomatisasi.
