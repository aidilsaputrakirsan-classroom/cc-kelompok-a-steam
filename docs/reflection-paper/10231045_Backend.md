## Arsitektur dan Keputusan Teknis

Sebagai Lead Backend dalam proyek ini, tanggung jawab utama saya adalah merancang logika sistem, pengelolaan database (basis data), dan memastikan komunikasi antar-layanan berjalan dengan baik. Transisi dari arsitektur *monolith* (sistem satu kesatuan) ke *microservices* (membagi aplikasi menjadi beberapa layanan mandiri) melahirkan beberapa keputusan teknis penting:

1. **Pemilihan FastAPI:** Kami memilih FastAPI karena mendukung proses asinkron (*asynchronous*) bawaan secara langsung. Karena aplikasi kami memanggil layanan luar yang butuh waktu lama untuk merespon—seperti Google Gemini untuk OCR/Ringkasan dokumen dan HuggingFace untuk membuat gambar—proses asinkron memastikan server tidak macet (*blocking*) saat menunggu jawaban. Ini membuat server sanggup melayani banyak permintaan pengguna secara bersamaan (*concurrency*) jauh lebih cepat dibanding framework lama seperti Flask.
2. **Penerapan Database per Service:** Kami memisahkan database menjadi `auth_db` (khusus data pengguna) dan `ai_db` (khusus riwayat AI). Dengan memisahkan database ini, jika tim frontend melakukan perubahan skema tabel pada layanan AI, tabel data pengguna di database sebelah tidak akan ikut terganggu. Ini juga menghindari kesalahan desain berupa *shared database* (database yang dipakai ramai-ramai), yang sering memicu error massal jika satu tabel bermasalah.
3. **Verifikasi Token Mandiri (Stateless JWT):** Untuk mengecek apakah pengguna sudah login, layanan AI kami tidak perlu bertanya secara langsung ke layanan Autentikasi lewat jaringan. Pemeriksaan ini dilakukan secara lokal di layanan AI menggunakan kunci rahasia bersama (*Shared Secret Key*). Cara ini sangat memangkas waktu loading (*latency*) dan menghemat beban jaringan (*network overhead*).

## Kesulitan Teknis dan Resolusi

Membagi aplikasi menjadi beberapa layanan mandiri ternyata memunculkan kendala baru di bagian ketahanan jaringan dan keamanan data.

1. **Mengatasi Efek Domino jika Layanan Mati (*Cascading Failure*):** Hambatan terbesar yang saya hadapi adalah bagaimana menjaga agar layanan AI tetap bisa dipakai jika layanan Autentikasi sedang down (mati). Awalnya, jika layanan Auth mati, fitur AI ikut crash total karena gagal memperbarui kuota penggunaan pengguna.
   Untuk mengatasinya, saya memasang sistem pengaman **Circuit Breaker** (pemutus sirkuit otomatis). Jika koneksi ke layanan Auth gagal beberapa kali secara berturut-turut, sirkuit akan terbuka dan memutus koneksi tersebut agar layanan AI tidak ikut crash. Sistem kemudian dialihkan ke mode alternatif (*degraded*), di mana pengguna tetap bisa membuat gambar/ringkasan, namun sistem hanya akan menyajikan data statistik sementara yang tidak memerlukan koneksi ke database Auth.
2. **Validasi Data Masuk (Input Validation):** Kami juga menemukan celah keamanan jika panjang karakter input tidak dibatasi. Menggunakan pustaka *Pydantic*, saya membatasi panjang password maksimal 128 karakter saat registrasi. Jika tidak dibatasi, penyerang bisa mengirimkan string password sepanjang jutaan karakter untuk membebani proses enkripsi password di CPU server kita (dikenal sebagai serangan *Denial of Service* atau DoS).

## Pelajaran Berharga

Membangun bagian backend untuk proyek ini memberikan saya banyak pelajaran baru yang sangat berguna:
1. **Mendesain Sistem yang Siap Error (*Resilience*):** Dalam arsitektur microservices, kita harus selalu berasumsi bahwa koneksi antar-layanan bisa terputus kapan saja. Membuat sistem yang tetap bisa melayani sebagian fungsi saat terjadi gangguan (*graceful degradation*) jauh lebih penting daripada berharap jaringan akan selalu aman 100%.
2. **Komunikasi Lewat Kontrak API:** Agar tim frontend (Incha) dan DevOps (Jetro) tidak bingung saat menyambungkan layanan, pembuatan dokumen **API Contract** (spesifikasi rute) dan penyediaan **Swagger UI** (halaman uji coba API otomatis) sangat membantu mempercepat pekerjaan dan mencegah miskomunikasi.
3. **AI sebagai Asisten, Bukan Pengganti Keahlian:** Pustaka AI sangat membantu mempercepat penulisan kode awal. Namun, pemecahan masalah yang rumit—seperti pengaturan sirkuit pengaman, pengamanan database, dan perapian struktur kode—tetap memerlukan pemahaman mendalam dari diri kita sendiri.

## 📖 Glosarium Istilah Teknis

*   **Monolith:** Arsitektur di mana semua fitur aplikasi digabung dalam satu program besar yang sama.
*   **Microservices:** Memisahkan satu aplikasi besar menjadi beberapa aplikasi kecil yang mandiri (misal: memisahkan layanan User dan layanan AI).
*   **Asynchronous:** Metode kerja server yang bisa mengerjakan tugas lain tanpa harus menunggu tugas sebelumnya selesai.
*   **Concurrency:** Kemampuan server untuk menangani banyak request pengguna secara bersamaan tanpa membuat server melambat.
*   **Stateless JWT:** Token keamanan terenkripsi yang berisi data pengguna, sehingga server bisa memverifikasi status login tanpa perlu memeriksa database setiap waktu.
*   **Shared Secret Key:** Kunci rahasia yang disimpan bersama di beberapa layanan agar dapat membaca data terenkripsi yang sama.
*   **Latency / Overhead:** Waktu tunda respon server dan beban tambahan jaringan yang membuat aplikasi melambat.
*   **Cascading Failure:** Kegagalan berantai di mana matinya satu komponen sistem menyebabkan komponen lainnya ikut mati.
*   **Circuit Breaker:** Pola pengaman untuk mendeteksi error koneksi antar-layanan, lalu otomatis memutus aliran request ke layanan yang rusak agar sistem utama tidak ikut crash.
*   **Degraded Mode:** Kondisi darurat di mana aplikasi tetap bisa berjalan melayani sebagian fitur utama meskipun beberapa layanan pendukung sedang mati.
*   **Denial of Service (DoS):** Serangan siber untuk membuat server kehabisan daya (CPU/Memori) sehingga tidak bisa melayani pengguna asli.
